# TinaCMS Setup & Deployment Guide

This document explains the TinaCMS configuration for the olliewritesthings.com site (Astro + GitHub Pages).

## Overview

The site uses **TinaCMS** to provide a visual content management interface for managing:
- **Blog Posts** - Located in `src/content/blog/` with fields: title, description, pubDate, updatedDate, heroImage
- **Pages** - Located in `src/content/pages/` with fields: title, intro

The admin interface is available at `/admin/` on the live site and in local development.

## Current Configuration

### tina/config.ts

The configuration file has been updated to:

1. **Use environment variables** for sensitive credentials
   - `NEXT_PUBLIC_TINA_CLIENT_ID` - TinaCloud client ID (public, safe to expose)
   - `TINA_TOKEN` - TinaCloud authentication token (private, must be protected)

2. **Manage Astro's actual content directories**
   - Blog collection: `src/content/blog/`
   - Pages collection: `src/content/pages/`
   - Removed old `content/posts/` collection that was causing schema mismatch

3. **Define proper content schemas**
   - Blog posts support: title, description, pubDate, updatedDate, heroImage, body
   - Pages support: title, intro, body

### GitHub Actions Workflow (.github/workflows/deploy.yml)

The deployment workflow now:

1. Installs dependencies and builds TinaCMS admin assets (if credentials are configured)
2. Builds the Astro site
3. Copies admin assets to the build output
4. Deploys everything to GitHub Pages

If credentials aren't configured, a helpful fallback page is created at `/admin/` with setup instructions.

## Enabling TinaCMS

### Step 1: Set Up TinaCloud Account

1. Go to [tina.io](https://tina.io)
2. Sign up for a free account
3. Create a new organization/project
4. Connect your GitHub repository

### Step 2: Get Your Credentials

From your TinaCloud dashboard:

1. Note your **Client ID**
2. Generate and copy your **Auth Token**

### Step 3: Configure GitHub Actions Secrets

Add these as **Repository Secrets** in GitHub (Settings → Secrets and variables → Actions):

- `NEXT_PUBLIC_TINA_CLIENT_ID` → Your TinaCloud Client ID
- `TINA_TOKEN` → Your TinaCloud Auth Token

After adding secrets, the next deployment will build and deploy the full TinaCMS admin interface.

### Step 4: Local Development (Optional)

To test TinaCMS locally:

1. Add credentials to `.env` file in the repo root:
   ```
   NEXT_PUBLIC_TINA_CLIENT_ID=your_client_id
   TINA_TOKEN=your_token
   TINA_PUBLIC_IS_LOCAL=true
   ```

2. Run `npm run dev` to start the development server
3. Access the admin at `http://localhost:3000/admin/`

Note: `.env` is in `.gitignore` so credentials won't be committed.

## Building & Deploying

### Local Build
```bash
npm run build
# Generates:
# - dist/ → Astro site output
# - public/admin/ → TinaCMS admin interface (included in dist by Astro)
```

### Deploy to Production

Simply push to the `main` branch. GitHub Actions will:

1. Check for GitHub secrets
2. Build Tina admin (if credentials exist)
3. Build Astro site
4. Deploy to GitHub Pages at https://olliewritesthings.com

## Troubleshooting

### Admin page shows "TinaCMS Not Configured"

This is expected if `NEXT_PUBLIC_TINA_CLIENT_ID` and `TINA_TOKEN` aren't configured as GitHub secrets.

**Fix:** Add the secrets (see Step 3 above) and push a new commit to trigger a rebuild.

### Content changes don't appear on the site

1. Verify changes are saved in the TinaCMS editor
2. Wait for GitHub Actions deployment to complete
3. Clear browser cache (Ctrl+Shift+Delete / Cmd+Shift+Delete)

### Schema mismatch errors

If you see "GraphQL schema mismatch" warnings:

1. This is expected when credentials aren't configured
2. Once you add GitHub secrets and trigger a new build, it should resolve
3. The locally-generated schema in `tina/__generated__/` is used for local development

### Can't connect to TinaCMS in local development

1. Ensure `.env` has valid `NEXT_PUBLIC_TINA_CLIENT_ID` and `TINA_TOKEN`
2. Let's Encrypt certificate issues on localhost can sometimes prevent connection
3. Try using a different port or checking browser console for errors
4. As a workaround, use the deployed admin at https://olliewritesthings.com/admin/

## File Structure

```
tina/
  ├── config.ts              # Main TinaCMS configuration (edited)
  └── __generated__/         # Auto-generated by Tina CLI (from config.ts)
      ├── schema.gql         # GraphQL schema
      ├── client.ts          # Generated client code
      └── ... other generated files

public/
  └── admin/
      └── index.html         # TinaCMS admin interface (deployed to /admin/)

src/content/
  ├── blog/                  # Blog posts (managed by TinaCMS)
  │   ├── first-post.md
  │   ├── second-post.md

  └── deploy.yml             # GitHub Actions workflow (updated to build Tina)
```
